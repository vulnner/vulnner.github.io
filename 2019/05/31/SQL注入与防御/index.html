SQL注入与防御

Posted on 2019-05-31 | In Web安全 , SQL注入

大多注入攻击的本质，都是把用户输入的数据当做代码执行。因此这里需要两个关键条件：用户能够控制输入；原程序要执行的代码，拼接了用户的输入。这篇博客介绍几种SQL注入攻击和防御办法。

1.1 布尔盲注

很多时候，Web服务器关闭了错误回显，单靠加'可能判断不出来是否存在注入。为了应对这种情况，我们可以使用所谓的盲注。盲注分为时间盲注和布尔盲注。

所谓“盲注”，就是在没有错误回显时完成的注入攻击。由于关闭了错误回显，我们不知道注入的SQL语句是否可以得到执行，因此我们可以使用盲注，观察页面是否发生延迟或变化，来判断语句是否得到执行。

以上，如果and 1=2得到了执行，返回的页面应该为空或者是出错页面。

在这个过程中，虽然关闭了错误回显，但是我们仍然能够通过简单的条件判断，然后比较页面返回结果的差异，从而判断SQL注入漏洞是否存在。

比如使用benchmark(count, expr)或sleep(s),如果页面延迟显示，则表示该函数执行成功，说明存在SQL注入。

3 防御SQL注入

前面介绍了一些SQL注入的攻击技巧，下面从防御的角度来看，要做的事情有两件：

1） 找到所有的SQL注入漏洞

2）修补这些漏洞

SQL注入的防御并不简单，开发者常常走入一些误区。比如有的开发者觉得只要转义掉一些特数字符，或者过滤一些单词，就可以防止SQL注入了。但这是不够的。

  • 比如开发者使用$id = msql_real_escape_string($_GET['id'])，当然还有开发者喜欢用addslashes()。

    可是，msql_real_escape_string()这个函数只会转义' " \r \n NULL Ctrl-Z，addslashes()只转义双引号，只要payload不用到这些字符，很容易就能绕过。

  • 还有的喜欢单纯的过滤某些单词，比如“空格”，“括号”等特殊字符，还有一些SQL保留字，如select、insert等。

    这种基于黑名单的过滤，一般都会存在一些问题。

    • 不需要空格：select/**/flag/**/from/**/flag
    • 不需要括号、引号：select passwd from users where user=0x61646D696E
